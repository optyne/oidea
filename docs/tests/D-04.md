# D-04 檔案附件欄位 — 測試文件

> 功能 ID：**D-04**
> 模組：P5 文件與結構化資料
> 依賴：D-02 (結構化資料庫)、既有 `File` model (from `files/` 模組)
> 實作檔案：`backend/src/databases/` (於 D-02 基礎上擴充)
> 測試檔案：`backend/src/databases/databases.service.spec.ts`
> 最後更新：2026-04-20

---

## 1. 功能範圍

在資料庫 (D-02) 裡新增 `file` 型別的欄位，每列該欄位可掛 1 個或多個既有 `File`（來自現有 FilesModule 上傳的檔案）。主要用途：合約 PDF、設計稿、請款單附件等。

| 能力                                        | 支援 |
| ------------------------------------------- | ---- |
| Column type = `file`                        | ✅   |
| `options.multiple` 預設 false，true = 多檔 | ✅   |
| `options.accept`（MIME 白名單提示欄位）    | ⚠️ 目前僅驗 schema，實際過濾留給 UI |
| Cell value schema `{ fileIds: string[] }`  | ✅   |
| 寫入時驗證 fileId 存在、未軟刪、同 workspace | ✅   |
| 檔案上傳本身                                | ⏳ 由既有 `POST /files/upload` 負責；D-04 不重做 |
| 下載 / 簽章網址                             | ⏳ 既有 FilesService 負責 |
| File 被刪除時自動清掉 cell 的 reference     | ❌ 目前 cell 保留 id（fallback：讀取時發現 deletedAt 再標 stale） |

---

## 2. Data 模型 (不新增 Prisma 表)

D-04 **不改 schema**。重用：
- `DatabaseColumn.type = 'file'`，`options` JSON：`{ multiple?: boolean, accept?: string[] }`
- `DatabaseCell.value` JSON：`{ fileIds: string[] }`
- `File` 既有模型（`workspaceId` / `deletedAt` / `url` 等）

> 設計取捨：JSON 儲存，不拉 join table，換來變更成本低，代價是 no FK integrity。依賴 service 層驗證。

## 4. 測試案例 (TC)

> 編號規則：TC-D04-00X = column、TC-D04-01X = cell (addRow)。

### 4.1 Column 建立

| ID         | 情境                            | 輸入                             | 預期                        |
| ---------- | ------------------------------- | -------------------------------- | --------------------------- |
| TC-D04-001 | file 欄位無 options             | `{type:'file'}`                  | 建立成功 (預設單檔)         |
| TC-D04-002 | file + multiple=true            | `{options:{multiple:true}}`      | 建立成功                    |
| TC-D04-003 | multiple 非 boolean             | `{options:{multiple:'yes'}}`    | `BadRequestException`       |
| TC-D04-004 | accept 非陣列                   | `{options:{accept:'pdf'}}`      | `BadRequestException`       |

### 4.2 Cell (addRow with file cell)

| ID         | 情境                                 | 前置檔案狀況                          | 輸入 `values['c-f']`            | 預期                  |
| ---------- | ------------------------------------ | ------------------------------------- | ------------------------------ | --------------------- |
| TC-D04-010 | 同 workspace 合法檔案                | `file.findMany` → `[{f-1, WS_ID}]`    | `{fileIds:['f-1']}`            | 建立成功              |
| TC-D04-011 | value 型別錯誤                       | —                                     | `'f-1'`                        | `BadRequestException` |
| TC-D04-012 | fileIds 元素非字串                   | —                                     | `{fileIds:[123]}`              | `BadRequestException` |
| TC-D04-013 | 單檔欄位塞 2 個檔                    | —                                     | `{fileIds:['f-1','f-2']}`      | `BadRequestException` |
| TC-D04-014 | multiple=true 多檔                   | `[{f-1,WS},{f-2,WS}]`                 | `{fileIds:['f-1','f-2']}`      | 建立成功              |
| TC-D04-015 | 不存在 / 已軟刪                      | `file.findMany` → `[]`                | `{fileIds:['ghost']}`          | `BadRequestException` |
| TC-D04-016 | 檔案屬於他 workspace                 | `[{f-1, 'other-ws'}]`                 | `{fileIds:['f-1']}`            | `BadRequestException` |
| TC-D04-017 | 空陣列（清空附件）                   | —                                     | `{fileIds:[]}`                 | 建立成功、不查 DB     |

## 8. 已知限制

1. **孤兒 file reference**：`File` 被刪後，cell.value 仍保留其 id，讀取時需 UI 層過濾 `deletedAt`。
2. **Accept 未強制**：`options.accept` 僅驗格式 (array)，實際 MIME 檢查在上傳階段或前端做。
3. **沒有批次掛檔 API**：要掛 N 個檔仍需發 N-append 的 `updateRow`（整陣列覆蓋）。
4. **無 attach/detach helper**：直接覆蓋整個 `fileIds`；競態由後進者勝。
