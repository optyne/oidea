## 1. 功能範圍

讓使用者一次把同樣內容發到多個頻道 — 對應員工工作流「把整理好的資料一次丟到不同工作群組」。

| 能力                                                   | 狀態 |
| ------------------------------------------------------ | ---- |
| 一次 1–50 個頻道                                       | ✅   |
| channelIds 自動去重                                    | ✅   |
| 所有目標須為同一 workspace                             | ✅   |
| 任一頻道非成員 → 整批拒絕 (Forbidden)                  | ✅   |
| 任一頻道不存在 → NotFound                              | ✅   |
| 原子寫入 (Prisma `$transaction`)                       | ✅   |
| 每筆訊息共用 `broadcastId` (uuid)                      | ✅   |
| WebSocket `newMessage` 事件逐頻道推送                  | ✅   |
| 同步更新 / 撤回一整組廣播                              | ❌（日後可掃 `broadcastId` 群組處理）|
| 排程發送 (C-17 會接)                                   | ❌   |

---

## 2. Data 模型變更

`Message` 多一欄 `broadcastId String?` + `@@index([broadcastId])`；不新增表。
查詢「某 broadcast 的所有訊息」：`where: { broadcastId: ... }`。

---

## 3. API

```
POST /messages/broadcast
Authorization: Bearer <jwt>
{
  "channelIds": ["c-1", "c-2"],
  "content": "月底結算公告",
  "type": "text",             // 可省略，預設 text
  "metadata": { ... }         // 可省略
}
→ 200 { "broadcastId": "<uuid>", "messages": [ <Message>, <Message> ] }
```

錯誤碼：
- 400 空陣列 / 跨 workspace / content 過長 / 超過 50 個頻道
- 403 非頻道成員（訊息附上哪個頻道）
- 404 部分頻道不存在

## 9. 已知限制

1. **編輯 / 撤回 scope** — 現階段改原訊息只影響該頻道那一筆；若要「一改全改」需後續做 `broadcast revision`。
2. **順序不保證** — 多頻道 `createdAt` 幾乎同時，但不保證完全同時；前端聚合顯示要容忍 ±ms 誤差。
3. **WS emit 後失敗** — 訊息已寫入 DB 但某個房間 emit 噴錯不會 rollback；這符合常見 WebSocket 設計 (at-most-once broadcast)。
4. **上限 50** — 硬上限避免濫用群發；企業級若要更高需加節流 / 限流層。
5. **@mentions 尚未抽取** — 廣播訊息不自動建立 `Mention` 紀錄（現有 `create` 路徑也沒做，屬既有限制）。
